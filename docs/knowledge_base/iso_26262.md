## 1. What is ISO 26262?

**Full title:** "Road vehicles — Functional safety"
**First edition:** 2011 (passenger cars only). **Second edition:** 2018 — extended to all road vehicles except mopeds; expanded from 10 to 12 parts.
**Root standard:** Adapted from IEC 61508 (generic functional safety), tailored for automotive E/E systems.

**Core purpose:** Manage hazards caused by *malfunctioning* electrical and electronic systems. Does NOT address intentional attacks (→ ISO/SAE 21434) or functional insufficiency of correctly-functioning systems (→ ISO 21448 SOTIF).

**Practical mandate:** Voluntary standard, but effectively mandatory — EU/Japan/South Korea type-approval authorities expect compliance; OEMs impose it contractually; product liability frameworks treat it as the standard of care.

## 3. ASIL — Automotive Safety Integrity Levels

**Key rule: ASIL is assigned to safety goals, not to components.**

| Level | Risk / Rigor | Typical examples |
|---|---|---|
| **QM** | Standard quality management; ISO 26262 lifecycle not triggered | Climate control, infotainment |
| **ASIL A** | Lowest safety-critical rigor | Tail lights, horn |
| **ASIL B** | Increased independent verification, stricter coding | Headlights, brake lights |
| **ASIL C** | Semi-formal methods recommended | Adaptive cruise control (driver supervisory) |
| **ASIL D** | Highest rigor; formal notation, MC/DC, tightest HW metrics | Airbag deployment, brake-by-wire, EPS |

### ASIL Determination — HARA Parameters (Part 3)

Three parameters, each on an ordinal scale:

| Parameter | Scale | Meaning |
|---|---|---|
| **Severity (S)** | S0–S3 | S0=no injury; S1=light/moderate; S2=severe/life-threatening; S3=fatal |
| **Exposure (E)** | E0–E4 | E0=incredibly unlikely; E4=high probability / routine driving condition |
| **Controllability (C)** | C0–C3 | C0=generally controllable; C3=difficult or impossible to control |

**ASIL anchor rule:** S3 + E4 + C3 = **ASIL D**. Any single parameter at level 0 yields QM. All other combinations map to ASIL A, B, or C via lookup table.

**Example:** Unintended motor torque while stationary in urban traffic = S3 (fatal collision) + E4 (routine signaling) + C3 (no driver reaction time) = **ASIL D**.

### ASIL vs. Other Integrity Level Schemes

| Scheme | Standard | Scale | Domain | Basis |
|---|---|---|---|---|
| ASIL | ISO 26262 | QM, A–D | Automotive | Qualitative S/E/C |
| SIL | IEC 61508 | 1–4 | Industrial/machinery | Quantitative failure rate |
| DAL | DO-178C / ARP4754A | A–E | Aerospace | Qualitative |

**No normative mapping exists** between ASIL and SIL — a SIL 3 certificate cannot be claimed as ASIL D without separate automotive evidence.

## 4. Safety Lifecycle — V-Model Structure

The ISO 26262 lifecycle follows the V-model across Parts 3–6. Left arm = requirements and design; right arm = verification and validation.

```
Part 3: Item Definition
Part 3: HARA → Safety Goals
Part 3: Functional Safety Concept (FSC)
    Part 4: Technical Safety Concept (TSC)
        Part 4: System Design / Safety Requirements
            Part 5: HW Design          Part 6: SW Requirements
                Part 5: HW Unit        Part 6: SW Architecture
                                        Part 6: SW Unit Design/Implementation
                Part 5: HW Integration Test      Part 6: SW Unit Test
            Part 4: System Integration Test      Part 6: SW Integration Test
        Part 4: System Verification
    Part 4: Item Validation (vehicle level)
Part 7: Production, Operation, Service, Decommissioning
```

Part 8 (supporting processes) operates continuously across all phases.

## 5. Part 3 — Concept Phase in Detail

### Step 1: Item Definition
Precisely scopes the system being analyzed. Must document:
- Functionality, interfaces with other systems
- Operational conditions and modes
- Regulatory requirements
- System boundary
- Preliminary architecture

### Step 2: HARA (Hazard Analysis and Risk Assessment)
Systematic identification of hazardous events = operational situation × malfunction.
Each hazardous event receives S/E/C ratings → ASIL assignment → **Safety Goal**.

**Safety Goals** are vehicle-level behavioral requirements (implementation-independent, negative/preventive form):
- Example: *"Prevent unintended traction torque exceeding X Nm while vehicle is stationary"* (ASIL D)
- Each safety goal inherits the ASIL of the hazardous event that generated it

**Fault Tolerant Time Interval (FTTI):** Maximum time from fault occurrence to potential hazardous event — defines the timing budget for all downstream safety mechanisms.

### Step 3: Functional Safety Concept (FSC)
Bridges safety goals to preliminary architecture. Defines:
- **Functional safety requirements (FSRs)** — how goals will be achieved (fault prevention, detection, safe state transition, fault tolerance)
- **Safe states** — defined system states where the hazardous event cannot occur (e.g., "engine torque = 0")
- **Degradation strategies** — controlled reduction of functionality upon fault detection
- **Allocation** — which system element is responsible for which FSR

## 6. Part 4 — System Level Development

**Technical Safety Concept (TSC):** Refines FSRs into technical safety requirements (TSRs) with specific implementation constraints. Allocates requirements to hardware and software elements.

**Key analyses at system level:**
- **Dependent Failure Analysis (DFA):** Confirms that elements claimed as independent do not share common causes (power supplies, clock sources, physical proximity, shared software modules)
- **Safety Analysis (FMEA / FTA):** Identifies potential failure modes and their effect on safety goals; determines diagnostic coverage requirements

## 7. Part 5 — Hardware Development

### Quantitative Hardware Metrics

Three metrics demonstrate hardware safety; values are ASIL-dependent:

| Metric | Meaning | ASIL B | ASIL C | ASIL D |
|---|---|---|---|---|
| **SPFM** — Single Point Fault Metric | Coverage of faults causing immediate safety violation (no protection) | >=90% | >=97% | >=99% |
| **LFM** — Latent Fault Metric | Detection of dormant faults before a second fault triggers the hazard | >=60% | >=80% | >=90% |
| **PMHF** — Probabilistic Metric for Hardware Failures | Residual random hardware failure rate (per hour of vehicle operation) | <1e-7/h | <1e-7/h | <1e-8/h |

FMEDA (Failure Mode, Effects, and Diagnostic Analysis) is the primary method for demonstrating these metrics.

### Hardware Safety Mechanisms
Examples: CRC checks, redundant sensors, watchdog timers, voltage monitoring, plausibility checks between redundant channels.

## 8. Part 6 — Software Development

### Software Safety Requirements by ASIL

| Requirement | ASIL A | ASIL B | ASIL C | ASIL D |
|---|---|---|---|---|
| **Notation** | Informal natural language | Natural language | Semi-formal (SysML, Stateflow) recommended | Semi-formal effectively required |
| **Coding guideline** | Language subset required | Language subset required | MISRA-level enforcement | MISRA-level enforcement mandatory |
| **Structural coverage — statement** | Required | Required | Required | Required |
| **Structural coverage — branch** | Required | Required | Required | Required |
| **Structural coverage — MC/DC** | Not required | Not required | Recommended | **Required** |
| **Independence of testing** | Basic | Separate team | Separate team | Independent organization |

**MC/DC (Modified Condition/Decision Coverage):** Each Boolean condition in every decision must independently affect the decision outcome. Requires dedicated test cases for each condition and is the most demanding structural coverage criterion.

**MISRA C:2012 / MISRA C++ 2023:** De facto automotive coding standards. ISO 26262 requires a demonstrable language subset reducing systematic faults — MISRA compliance via static analysis (Polyspace, LDRA, Parasoft) is the dominant implementation path.

### Key Software Work Products (Part 6)
- Software safety requirements specification
- Software architectural design (static/dynamic aspects)
- Software unit design and implementation
- Software unit verification report (coverage evidence)
- Software integration test specification and report
- Software qualification test specification and report

### Software Safety Mechanisms
Examples: defensive programming (invariant checks), range checks on inputs, end-to-end protection (E2E profiles), memory protection (MPU), timing monitoring (watchdog), software redundancy with independent voter.

## 9. Part 8 — Supporting Processes

### Requirements Management
Bidirectional traceability from safety goals through FSRs → TSRs → SW requirements → test cases → test results. Every requirement checked for completeness, consistency, verifiability, clarity. AutoPragma implements this as FR-001 (Traceability Engine).

### Configuration Management
Version control, baseline management, change tracking, access control on all safety-related work products: source code, design files, test results, safety documentation. AutoPragma implements this as FR-009 (Baseline Management).

### Change Management
Every modification assessed for functional safety impact before approval. Safety impact analysis documented and traceable. AutoPragma implements this as FR-007 (Gate Automation) + FR-012 (Audit Trail).

### Documentation Management
ISO 26262 is fundamentally evidence-based. Every safety activity must produce a documented work product. Documentation is the safety case.

### Tool Qualification (Part 8 Clause 11)

**Two-step process:**
1. **Tool Classification** — determine whether qualification is needed
2. **Tool Qualification** — establish required confidence (if needed)

**Classification dimensions:**

| Dimension | Level | Meaning |
|---|---|---|
| **Tool Impact (TI)** | TI1 | Tool output not used to generate or verify safety requirements → no safety impact |
| | TI2 | Tool output could introduce or fail to detect errors in safety-related items |
| **Tool Error Detection (TD)** | TD1 | High confidence that downstream activities detect tool errors |
| | TD2 | Medium confidence |
| | TD3 | Low confidence |

**Tool Confidence Level (TCL) matrix:**

| | TD1 | TD2 | TD3 |
|---|---|---|---|
| **TI1** | TCL1 | TCL1 | TCL1 |
| **TI2** | TCL1 | TCL2 | TCL3 |

- **TCL1:** Standard quality processes sufficient — no formal qualification required
- **TCL2:** Formal qualification required (medium confidence)
- **TCL3:** Rigorous qualification required (highest confidence; most demanding for ASIL C/D)

**Qualification methods (for TCL2/TCL3):**

| Method | Approach |
|---|---|
| **1a** — Increased confidence from use | Evidence of successful prior use of same tool version in comparable safety context |
| **1b** — Evaluation of development process | Audit of tool vendor's quality-assured development process |
| **1c** — Validation of software tool | Dedicated test suite validating tool behavior (most direct; widely used for compilers) |
| **1d** — Development per safety standard | Tool itself developed under IEC 61508 or equivalent |

**Cost-reduction strategy:** Improving downstream verification activities (adding independent static analysis, unit testing) can shift a tool from TCL3 to TCL1 by increasing TD1 confidence, eliminating formal qualification.

**AutoPragma relevance:** The AutoPragma platform itself must be classified per TCL. If it generates or gates safety work products (FR-007, FR-009), it likely classifies as TI2. Improving downstream human review gates (FR-008) and audit trail (FR-012) can push TD toward TD1, reducing qualification burden. See SRS NFR-005.

### Additional Part 8 Scopes

**Software Component Qualification (Clause 12):** Pre-existing components (COTS libraries, OSS, reused modules) assessed for safety-related reuse based on development process, failure modes, existing verification evidence.

**Proven in Use (Clause 14):** Field operation evidence used to argue safety — requires comparable operational context, sufficient volume/duration, systematic failure analysis, absence of safety-relevant failures.

## 10. Part 9 — ASIL Decomposition

**Purpose:** Permit high-ASIL requirements to be implemented as two *sufficiently independent* lower-ASIL sub-requirements, reducing development cost while maintaining item-level safety goals.

### Valid Decomposition Mappings

| Parent ASIL | Valid decomposition pairs |
|---|---|
| **ASIL D** | D(D)+QM(D) · C(D)+A(D) · B(D)+B(D) |
| **ASIL C** | C(C)+QM(C) · B(C)+A(C) |
| **ASIL B** | B(B)+QM(B) · A(B)+A(B) |

**Notation:** ASIL B(D) = "developed to ASIL B rigor for an originally ASIL D requirement." The original safety goal's ASIL never changes — only the implementation rigor for each sub-element does.

### Independence Requirement
Decomposition is only valid when sub-elements are **sufficiently independent** — sharing any of the following invalidates the decomposition:
- Same microprocessor / silicon die
- Common power supply or clock source
- Common software module or OS
- Same design team or reviewer (for higher ASIL)
- Physical proximity creating single-point failure (e.g., thermal runaway)

**Dependent Failure Analysis (DFA)** must explicitly confirm independence for each decomposition claim.

## 11. Safety Element Out of Context (SEooC) — Part 10

Pre-qualified components (microcontroller safety blocks, motor drivers, software stacks) developed without knowing the final integration context.

**Supplier obligations:**
- Define **safety assumptions** about operating conditions: voltage range, temperature, frequency, response times, functional interfaces
- Produce safety case conditional on those assumptions

**Integrator obligations:**
- Validate that actual deployment conditions match the supplier's stated assumptions
- Document any assumption mismatches as safety-relevant deviations requiring analysis

**Common failure mode:** Supply voltage ripple, temperature profiles, or ECU response timing in the actual vehicle differs from supplier assumptions — creates ASIL gap that is often only discovered during system validation.

## 12. Relationship to Other Standards

| Standard | Relationship to ISO 26262 |
|---|---|
| **ISO/SAE 21434** | Complementary — 26262 covers accidental failures; 21434 covers intentional cyber attacks. Both required; HARA and TARA often run concurrently and cross-feed. |
| **ISO 21448 (SOTIF)** | Covers functional insufficiency of correctly-functioning systems (e.g., sensor limitations in ADAS). Neither 26262 nor 21434 addresses this. |
| **ASPICE** | Process quality complement. ASPICE SWE.1–SWE.6 maps to Part 6 software lifecycle. ASPICE CL3 process institutionalization supports Part 2 safety management obligations. |
| **IEC 61508** | Parent standard. ISO 26262 is automotive-specific adaptation. |
| **MISRA C/C++** | De facto coding standard referenced by Part 6. Polyspace is the dominant compliance verification tool. |

## 13. Common Implementation Pitfalls

1. **Controllability rating inflation** — claiming C1 instead of C3 without supporting evidence (driving studies, simulation data)
2. **Decomposition without independence** — splitting ASIL D across two software paths on the same MCU without hardware-level separation
3. **HARA as one-time artifact** — failing to revisit HARA when integration testing reveals new hazardous events or architectural changes alter operating conditions
4. **Traceability breaks** — OEM → Tier 1 → Tier 2 requirement chains disconnect during mid-program redesigns; breaks create safety case gaps
5. **Tool qualification neglect** — new tools adopted mid-project (compilers, code generators, static analyzers) without classification and qualification
6. **Unrealistic FMEDA coverage claims** — 99% diagnostic coverage across all fault modes without analysis of how each coverage mechanism actually works signals analysis gaps
7. **SEooC assumption mismatches** — supplier-stated voltage/temperature/timing assumptions not validated against actual integration environment
8. **Retrofit safety cases** — attempting to build a safety case retrospectively for a completed product is expensive and leaves audit gaps; safety case must be built incrementally with the product
