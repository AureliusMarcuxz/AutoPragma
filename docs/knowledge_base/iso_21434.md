## 1. What is ISO/SAE 21434?

**Full title:** "Road vehicles — Cybersecurity engineering"
**Published:** August 2021, jointly by ISO and SAE International
**Scope:** All road vehicles and their E/E systems — motorcycles, cars, trucks — developed or modified after release. Does NOT prescribe specific technologies; defines what to achieve, not how.

**Why it exists:** Modern vehicles contain 100+ million lines of code across 100+ ECUs with attack surfaces via telematics, Wi-Fi, Bluetooth, USB, OBD-II, V2X, and OTA. ISO 26262 covers *accidental* failures; ISO/SAE 21434 covers *intentional* cyber attacks. They are complementary — both are required for modern vehicle compliance.

**Three core pillars:**
1. Organizational-level cybersecurity governance (CSMS)
2. Systematic security activities at project level (lifecycle)
3. Continuous monitoring and response post-launch

## 2. Relationship to UNECE R155 / R156

| Regulation | What it requires | Mandate |
|---|---|---|
| **UNECE R155** | Approved Cybersecurity Management System (CSMS) for vehicle type approval | Mandatory since July 2024 in 60+ countries (EU, Japan, Korea, etc.) |
| **UNECE R156** | Software Update Management System (SUMS) | Companion to R155 |
| **ISO/SAE 21434** | Engineering framework — defines *how* to do cybersecurity engineering | Voluntary standard; de facto evidence base for R155 certification |

**Practical flow:** OEMs need UNECE R155 type approval → they implement a CSMS → ISO/SAE 21434 work products serve as the engineering evidence for that CSMS.

## 6. TARA Methodology (Clause 15) — Detail

TARA is the centerpiece of ISO/SAE 21434. It is iterative and updates with new threat intelligence throughout the lifecycle.

### Step 1: Asset Identification
- Catalog cybersecurity-relevant assets: data (e.g., firmware, calibration data, user credentials), functions (e.g., OTA update, remote access), and interfaces (e.g., OBD-II, V2X, CAN, Ethernet)
- Define cybersecurity properties for each asset: **Confidentiality, Integrity, Availability, Authenticity (CIAA)**
- Define damage scenarios — what harm results if each property is violated

### Step 2: Threat Scenario Identification
- Map concrete attack vectors for each asset
- Use STRIDE framework: **S**poofing, **T**ampering, **R**epudiation, **I**nformation disclosure, **D**enial of service, **E**levation of privilege
- Supplement with threat databases: EMB3D, AutoISAC, NVD

### Step 3: Impact Rating
Rate severity of each damage scenario across four dimensions:

| Dimension | Considerations |
|---|---|
| **Safety** | Physical harm to persons (maps to ASIL concepts) |
| **Financial** | Economic loss to OEM, supplier, or customer |
| **Operational** | Vehicle functionality degradation |
| **Privacy** | Personal data exposure |

Scale: Negligible / Moderate / Major / Severe

### Step 4: Attack Feasibility Assessment
Evaluate attacker effort required using five factors:
1. **Time** — elapsed time to successfully carry out the attack
2. **Expertise** — level of knowledge required (layman → expert → multiple experts)
3. **Knowledge** — familiarity with the target item
4. **Opportunity** — access requirements (physical access vs. remote)
5. **Equipment** — tools and hardware needed

Output: Feasibility rating (Very Low / Low / Medium / High)

### Step 5: Risk Determination and Treatment
- Combine Impact × Feasibility → Risk Value
- Select risk treatment strategy:

| Treatment | Meaning |
|---|---|
| **Avoid** | Eliminate the asset or function creating the risk |
| **Reduce** | Implement cybersecurity measures to lower impact or feasibility |
| **Share** | Transfer risk contractually (insurance, supplier CIA) |
| **Retain** | Accept the residual risk with documented rationale |

Output: **Cybersecurity Goals** — top-level requirements derived from risks that must be reduced. Goals drive all downstream cybersecurity requirements (WP-09-03 → WP-10-01).

### TARA Scale Challenge
Single ECU: 20–50 assets, 100–300+ threat scenarios. Manual TARA takes 4–8 weeks per system. Automation tools are increasingly used (AI-assisted threat catalog matching, risk matrix computation).

## 7. Cybersecurity Case (WP-06-02)

The cybersecurity case is the overarching argument document demonstrating that the item achieves adequate cybersecurity. It links:

- Cybersecurity goals (WP-09-03) ← derived from TARA (WP-09-02)
- Cybersecurity concept (WP-09-05) ← implements goals
- Cybersecurity specifications (WP-10-01) ← refines concept to requirements
- Verification evidence (WP-10-05, WP-10-07) ← demonstrates requirements met
- Validation report (WP-11-01) ← demonstrates goals met at vehicle level

Structure mirrors an assurance case (claim → evidence → argument).

## 8. Supply Chain Obligations (Clause 7)

**Cybersecurity Interface Agreement (WP-07-01)** — mandatory formal document between every customer-supplier pair where cybersecurity-relevant work is distributed. Must specify:
- Which work products each party is responsible for
- Which TARA artifacts the supplier delivers with the component
- SLAs for vulnerability disclosure and response
- Criteria for evaluating supplier cybersecurity capability

**OEM responsibilities:**
- Establish and certify CSMS (R155 type approval)
- Evaluate supplier cybersecurity capability during sourcing
- Run cybersecurity validation at vehicle level (WP-11-01)
- Manage vulnerability information across the fleet lifecycle

**Tier 1 supplier responsibilities:**
- Deliver TARA, cybersecurity concept, and test evidence with component
- Maintain vulnerability monitoring for delivered components for 10–15 years
- Respond to vulnerability disclosures within CIA-agreed timelines
- Produce WP-07-01 for sub-tier suppliers

**Tier 2+ responsibilities:**
- Produce engineering evidence proportional to cybersecurity relevance of their component

## 9. ISO/SAE 21434 and ASPICE Integration

**ASPICE for Cybersecurity** extension published February 2022. Defines new process areas for cybersecurity assessment, covering:
- Cybersecurity requirements elicitation
- Cybersecurity implementation
- Risk treatment verification
- Risk treatment validation

**Integration strategy (avoid duplicate effort):**
- ASPICE incident handling → extend to cover cybersecurity incident response (ISO 21434 Clause 13)
- ASPICE change management (SUP.10) → extend to cover cybersecurity change impact (ISO 21434 Clause 8)
- ASPICE SUP.8 configuration management → extend to cover cryptographic key and certificate lifecycle
- ASPICE SWE.1 requirements analysis → extend to capture cybersecurity-tagged requirements with TARA traceability
- ASPICE SUP.1 QA records + ISO 21434 Clause 6 assessment → unified evidence record (see AutoPragma CONFLICT-01)

**Key principle:** Treat ISO 21434 cybersecurity activities as extensions of existing ASPICE process areas, not separate parallel processes. This is the AutoPragma design approach — JAMA and EA own cybersecurity artifacts; the platform enforces gates and traceability against them.

## 11. Common Audit Findings to Avoid

1. **Independence violations** — WP-06-03 assessment conducted by team reporting to project management (violates Clause 6.4.8)
2. **WP-05-05 failures** — organisational audit not performed by independent party
3. **TARA under-counting** — collapsing all 8 Clause 15 WPs into one artifact (produces 34–36 count instead of 42)
4. **Weak CIAs** — WP-07-01 lacks responsibility matrix and SLAs (common at Tier 1/Tier 2 boundary)
5. **Clause 8 fragmentation** — WP-08-01 through WP-08-06 merged into one spreadsheet, obscuring information flow and making audit trail non-traceable
6. **Incomplete CIAA dimensions** — impact ratings missing one or more of safety/financial/operational/privacy dimensions
